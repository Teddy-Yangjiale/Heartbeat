# Heartbeat Music Processor 改造目标与问题追踪

## 最终目标

用户在一个网页中上传心跳 WAV 与歌曲 WAV/MP3，完成心跳预处理、歌曲节拍/重拍/底鼓分析、S1/S2 保真编排、局部编辑、风格化自然混音，并导出 FLAC、MP3 或 WAV。网页在公开云端的安全模式下不因单个长任务长期占满内存，并为未来独立渲染 worker 保留清晰边界。

状态说明：`完成` 表示已有代码和自动化验证；`后续扩展` 表示不阻塞当前单机/Community Cloud 安全模式，但在高并发或高精度模型场景仍需要独立基础设施。

## A. 网页稳定性与资源问题

| 问题 | 根因 | 修改目标 | 状态 |
|---|---|---|---|
| 上传后内存翻倍 | `UploadedFile.getvalue()` 复制完整输入 | 使用 buffer view；文件大小和解码时长双重校验 | 完成 |
| 分析结果长期占用整首歌曲 | `song_analysis["audio"]` 放入 Session State | 只保存节拍、重拍、能量和最多 10,000 点波形概览 | 完成 |
| 心跳分析保留大量重复数组/文件/ZIP | 完整预处理结果进入 Session State | 增加 `web` 精简产物，只保留 4 个短试听 WAV、多周期素材池和必要报告字段 | 完成 |
| 最终渲染同时保存 4 个 WAV 和 ZIP 字节 | 所有产物在内存中编码并驻留 | 默认只生成最终混音；写入临时目录；下载点击时延迟读取 | 完成 |
| 中间轨数组过多 | 歌曲、duck、mix、master 多次复制 | 歌曲增益、低频 duck 和母带改为受控原地处理；分轨副本按需创建 | 完成 |
| 多用户同时渲染导致资源叠加 | Streamlit 同进程并发 | Community Cloud 安全模式使用全局单任务锁 | 完成 |
| 临时文件泄漏 | 会话异常退出可能不执行清理 | 清除任务时删除；启动/重跑时回收超过 6 小时的任务目录 | 完成 |
| 崩溃前缺少资源证据 | 无阶段内存数据 | 报告和日志记录解码、心跳渲染、母带、导出前 RSS | 完成 |
| 本地测试使用了错误 Python | 默认 Anaconda base 缺依赖 | 文档和批处理统一使用 `heartbeat` Conda 环境 | 完成 |
| 真正多用户并发 | Community Cloud 不适合 CPU 密集长任务 | UI + 对象存储 + 队列 + 独立 worker | 后续扩展 |

## C. 歌曲分析和音乐语义

| 问题 | 根因 | 修改目标 | 状态 |
|---|---|---|---|
| 只知道 BPM，不知道小节 | 第一检测 beat 被当成小节第一拍 | 推断 downbeat、显示置信度，并支持手动第一重拍和拍号 | 完成 |
| 全曲固定 BPM 误伤变速/停顿 | 全局中位速度 | 保留动态 beat grid；缺拍只在局部速度稳定且有 onset 支持时修复 | 完成 |
| intro/verse/chorus 听感一样 | 没有局部能量 | 为每个 beat 计算紧凑能量/低频重音；自动模式在低能量段疏化并降低力度 | 完成 |
| 重模型直接压垮网页 | Beat This/Demucs 计算和模型内存较大 | 云端安全模式使用 librosa + 手动校正；重模型放独立 worker | 后续扩展 |

| 问题 | 根因 | 修改目标 | 状态 |

## G. 验收标准

- 报告中 `maximum_anchor_alignment_error_ms <= 1 ms`，并明确模型节拍与 guide pulse 数量。
